# PRD 草稿：都江堰治水题材解谜小游戏

> 文档状态：草稿（Draft）  
> 基于讨论日期：2026-06-26  
> 适用平台：主流浏览器 H5 → 微信小游戏  

---

## 1. 产品定位

### 1.1 一句话描述
一款面向主流浏览器与微信小游戏的 **国风水利解谜小品**，让玩家在 60–90 秒的短关卡中通过“摆放治水构件 → 放水观察 → 失败/顿悟 → 再试”的循环，主动领悟李冰“堵不如疏、顺势而为”的治水智慧。

### 1.2 产品定位
- **品类**：文化科普向轻解谜游戏（Puzzle / Simulation）。
- **风格**：水墨国风 + 2D 横板表现。
- **核心体验**：可顿悟的确定性解谜，而非硬核物理模拟。
- **目标平台**：
  - 第一阶段：主流浏览器 H5（直接打开，便于测试与传播）。
  - 第二阶段：微信小游戏（利用社交分享与排行榜裂变）。
- **付费设计**：
  - 不卖资源、不卖跳关、不付费跳过。
  - 收入来自激励视频广告（换取一次提示）与外观/主题付费。

### 1.3 目标用户
| 维度 | 描述 |
|------|------|
| 年龄 | 12–35 岁 |
| 平台 | 移动端浏览器、微信小游戏 |
| 游戏经验 | 休闲玩家，接触过物理解谜、放置类或文化题材小游戏 |
| 核心需求 | 碎片化时间获得「有意义的好玩」体验，愿意尝试有文化深度的小品 |
| 单局时长 | 60–90 秒 |

### 1.4 核心价值主张
- 把抽象的都江堰治水理念转化为可操作的建造-测试-顿悟循环。
- 通过零惩罚重试降低挫败感，通过“节俭通关”引导玩家寻找优雅解。
- 让玩家在反复试错中主动发现规律，而不是被动接受知识灌输。

## 2. 用户痛点

1. **传统文化游戏太说教、太枯燥**  
   市面多数文化题材游戏以答题、看介绍为主，玩家被动接受知识，缺乏参与感与情感共鸣。

2. **休闲解谜游戏缺乏意义感，玩完就忘**  
   三消、物理解谜好玩，但很少能留下文化记忆或让玩家产生“原来如此”的顿悟。

3. **失败惩罚太重，挫败感强**  
   传统解谜游戏失败一次就要从头再来，导致玩家在中途流失。

### 2.1 本游戏如何解决
- **没有独立教程关，整局即教学**：玩家在 L1 硬堵失败中自然领悟“堵不如疏”。
- **零惩罚重试**：重置、撤销、重试均无代价，鼓励实验。
- **短关卡 + 即时反馈**：60–90 秒一关，失败反馈在 3–8 秒内出现。
- **文化命名与碑廊解锁**：成功瞬间给出知识点命名，将玩法与文化记忆绑定。

## 3. MVP 范围

### 3.1 包含内容

| 模块 | MVP 内容 | 说明 |
|------|----------|------|
| **关卡** | L1 完整可玩 + L2 占位 | L1 承担“筑墙失败”教学；L2 显示「开发中，敬请期待」 |
| **核心解谜** | 编辑 → 放水 → 模拟 → 结算 | 含放置、旋转、删除、撤销、重置、状态机 |
| **构件系统** | 石墙、竹笼 | 覆盖“硬堵失败”与“分水疏导”两种机制 |
| **水流模拟** | 2D 粒子系统 | 固定时间步长，保证跨设备确定性；视觉用墨滴粒子 |
| **金钱资源** | 金钱上限 + 节俭判定 | 放置扣钱、删除返还、结算判定是否节俭 |
| **关卡选择器** | 卷轴/地图形式 | 显示 L1、L2 状态，支持点击进入 |
| **存档** | 本地优先 + 服务端同步 | 浏览器用 localStorage；微信小游戏用 wx 存储；Java 后端保存进度 |
| **叙事** | 最简叙事 | 每关进入、成功、失败各一句文案 |
| **提示** | 单层提示 | L1 失败 1–2 次后触发老河工提示 |
| **音效/美术** | 占位级 | 基础 UI 音效、水墨风格占位图；完整资源后续迭代 |

### 3.2 明确不包含在 MVP
- 8 关完整内容（仅保留 L1 + L2 占位）。
- 完整碑廊系统（先简化为通关成就文案）。
- 三层提示树、复杂剧情演出。
- 真 3D 运行时渲染（采用 3D 预渲染成 2D 精灵图）。
- 多人排行榜、社交裂变（仅预留数据字段）。

### 3.3 关卡设计（MVP）

#### L1「堵」—— 筑墙失败
- **教学目标**：让玩家意识到“硬堵不可行”。
- **机制**：
  - 河道从左向右流，村庄在右下侧。
  - 玩家获得若干石墙，第一反应是横向筑坝阻挡。
  - 石墙在压力超过阈值后 0.8–2 秒内倒塌，村庄在约 10 秒内被淹。
- **成功路径**：用竹笼竖放分水，将主流导向远离村庄的支流。
- **节俭阈值**：实际消耗金钱 ≤ 关卡配置值。

#### L2「分」—— 开发中占位
- 关卡选择器中显示节点，文案：「开发中，敬请期待」。
- L1 通关后，L2 节点状态变化（例如解锁预览或高亮），但仍不可进入。

## 5. 核心流程

### 5.1 整体循环

```text
启动 → 标题页 → 关卡选择器 → 进入 L1 → 叙事开场 → 编辑阶段 → 放水 → 模拟阶段 → 结算 → 重试/返回
```

### 5.2 L1 单关详细流程

1. **进入关卡**
   - 玩家从 LevelMapScene 点击 L1。
   - 加载 LevelConfig（地形、水源、村庄、库存、金钱上限、节俭阈值）。
   - 初始化 PuzzleRuntime：状态 = Editing，金钱 = 上限，库存按配置填充。
   - 播放老河工开场白 NarrativeOverlay。

2. **编辑阶段（Editing）**
   - 底部工具栏显示可用构件及剩余数量。
   - 玩家拖拽构件到河道，系统自动吸附 0.5 米网格。
   - 点击已放置构件旋转 90°（rotStep = (rotStep + 1) % 4）。
   - 拖拽已放置构件到删除区，返还库存与金钱。
   - 顶部金钱条实时显示余额与预览消耗。
   - 支持撤销（最多 20 步）与重置。

3. **放水（Editing → Simulating）**
   - 玩家点击「放水」按钮。
   - 校验：库存与金钱足够、布局合法。
   - 进入 Simulating 状态，锁定编辑，「放水」按钮置灰并显示「模拟中…」。

4. **模拟阶段（Simulating）**
   - 水源按流量曲线（base → peak → stable）生成 2D 墨滴粒子。
   - 粒子沿 x 轴从左向右运动，与构件包围盒碰撞后按 WaterInteraction 改变速度：
     - 石墙：x 方向反弹并衰减，对墙产生水势压力。
     - 竹笼：竖放时分流向上下（y 方向）；横放时阻挡或改变 x 方向。
   - 石墙水势 ≥ collapseThreshold 后进入 Collapsed 状态，不再阻挡水流。
   - 每帧统计进入村庄受击区域的粒子数，累计 villageHitCount。
   - 达到失败阈值或最大模拟时长后进入 Settling。

5. **结算阶段（Settling）**
   - 成功：村庄未被淹没 → 显示「暂时安全」。
   - 失败：
     - 村庄被淹 → 「村子仍被淹」。
     - 关键构件倒塌导致村庄被淹 → 「墙倒了」。
   - 计算 isFrugal = consumedMoney ≤ frugalMoney。
   - 更新关卡状态：attempts + 1，解锁 L2 预览状态。
   - 玩家选择「重试」返回 Editing，或「返回」回到关卡选择器。

### 5.3 状态转换图

```text
Editing ──[点击放水/布局合法]──> Simulating
Simulating ──[时间到/触发条件]──> Settling
Settling ──[点击重试]──> Editing
Settling ──[点击返回]──> LevelMapScene
Simulating ──[点击暂停]──> Paused
Paused ──[点击继续]──> Simulating
```

## 6. 技术方案

### 6.1 总体架构

```text
┌─────────────────────────────────────────────────────────┐
│  客户端：Cocos Creator 3.x（2D 项目）+ TypeScript          │
│  ├── 表现层：Sprite、2D 粒子、Shader、UI 预制体           │
│  ├── 玩法层：核心解谜系统、构件系统、水流模拟、金钱系统    │
│  └── 数据层：本地存档、关卡配置、用户状态                 │
├─────────────────────────────────────────────────────────┤
│  服务端：Java（推荐 Spring Boot）+ REST API                │
│  ├── 关卡配置 API（LevelConfig、BuildingBlock）           │
│  ├── 用户进度同步（LevelState、解锁状态）                 │
│  └── 成就/排行榜校验（预留）                              │
└─────────────────────────────────────────────────────────┘
```

### 6.2 客户端技术选型

| 层级 | 技术/方案 | 说明 |
|------|-----------|------|
| 引擎 | Cocos Creator 3.x | 官方主推，支持 2D/3D，可一键发布 H5 与微信小游戏 |
| 项目模式 | 2D 项目 | 使用 Sprite、2D 碰撞、2D 粒子系统；美术资源为 3D 预渲染的 2D 精灵图 |
| 语言 | TypeScript | 类型安全，便于与 Java 团队协作 |
| 渲染 | Cocos 2D 渲染器 + 自定义 Shader | 水墨流动、墨滴粒子、水面波纹 |
| 碰撞 | 内置 2D 碰撞系统（BoxCollider2D / PolygonCollider2D） | 不引入 Box2D，避免过度设计 |
| 输入 | Cocos 触摸/鼠标事件 | 拖拽、点击、多点触控 |
| UI | Cocos UI 系统 | 按钮、进度条、弹窗、滚动视图 |
| 状态管理 | 自定义状态机 + 事件总线 | PuzzleState、LevelState、Wallet 等 |

### 6.3 服务端技术选型

| 层级 | 技术/方案 | 说明 |
|------|-----------|------|
| 框架 | Spring Boot | Java 生态成熟，团队熟悉 |
| 协议 | RESTful JSON | 状态无感知，便于 H5 与小游戏共用 |
| 数据库 | MySQL / PostgreSQL + Redis | 持久化用户进度；Redis 缓存热数据 |
| 配置 | JSON/YAML 关卡配置 | 策划可通过后台或 Excel 导出为 JSON，服务端下发 |
| 部署 | Docker + Nginx | 容器化部署，静态资源 CDN 加速 |

### 6.4 关键模块映射（从策划文档到 2D 横板）

| 原文档概念 | 2D 横板实现 |
|------------|-------------|
| Vector3 / x-z 平面 | 替换为 Vector2 / x-y 平面；x 为水流方向，y 为上下方向 |
| rotStep 0–3 | 保留，构件在 2D 平面内旋转 |
| 竹笼/鱼嘴分 z 方向 | 映射为 y 方向（上下）分水 |
| OBB 碰撞 | 使用 Cocos 2D 碰撞盒，旋转时长宽互换 |
| 村庄受击区域 | 2D 触发区域（Trigger），统计进入粒子数 |
| 水面 Mesh + Shader | 2D 横板水面：背景层 + 动态波纹 Shader + 粒子叠加 |

### 6.5 水流模拟关键设计

- **确定性保证**：
  - 固定时间步长（如 1/60 秒），与渲染帧率解耦。
  - 禁用随机数或采用固定种子；湍流使用预计算噪声表。
  - 浮点运算统一精度，必要时使用定点数或截断小数位。
- **性能分级**（仅影响视觉密度，不影响判定）：
  - 高端机：600 粒子，60fps。
  - 中端机：400 粒子，30fps。
  - 低端机：200 粒子，优先保证交互帧率。
- **判定与渲染解耦**：
  - 逻辑层维护独立的 `villageHitCount` 与 `pressure` 计算。
  - 渲染层只负责把逻辑状态可视化，粒子数量多少不改变判定结果。

### 6.6 数据流

```text
1. 启动时：客户端从服务端拉取 LevelConfig、BuildingBlock 配置（可本地缓存）。
2. 进入 L1：客户端根据配置初始化 PuzzleRuntime、MoneyWallet、Inventory。
3. 编辑/模拟：所有逻辑在客户端本地运行，服务端不参与。
4. 结算后：客户端将结果（success/fail、isFrugal、attempts）上报服务端保存。
5. 关卡选择器：从服务端拉取最新 LevelState，显示解锁与成绩。
```

### 6.7 美术与音效（MVP 占位）

- **美术**：
  - 构件：Blender 低模预渲染为 2D 精灵图，水墨风格。
  - 水流：软边墨滴粒子 + 横板水面波纹 Shader。
  - UI：水墨卷轴、印章动画（悟/安/倒/再）。
- **音效**：
  - 放置：竹笼摩擦、石墙落地。
  - 放水：水声随流量变化。
  - 成功：古琴单音；失败：低沉水声 + 村民惊呼。

## 7. 主要风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| **水流模拟跨设备非确定性** | 同一布局在不同设备结果不同，破坏顿悟体验 | 固定时间步长、统一浮点精度、禁用真随机、判定与渲染解耦 |
| **低端机粒子数下降影响视觉但需保证判定一致** | 玩家觉得“粒子少了所以结果不同” | 判定逻辑使用独立计数器，粒子数量只影响表现 |
| **L1 无法让玩家 get 到“堵不如疏”** | 玩家硬堵失败后迷茫流失 | 控制石墙倒塌时间 0.8–2 秒、村庄 10 秒内被淹、失败 1–2 次后触发老河工提示 |
| **微信小游戏包体积/性能/审核** | Cocos 打包过大或素材不合规 | 预渲染资源用图集压缩、首包只含 L1、后续关卡远程加载、真机测试覆盖中低端机型 |
| **团队 Cocos Creator 3.x 学习成本** | Vue/Java 团队切换到游戏引擎有适应期 | MVP 只做 L1，先跑通完整场景再扩展；关键岗位提前做技术预研 |
| **商业模式与文化体验的冲突** | 广告/付费设计破坏“零惩罚、不卖资源”红线 | 仅激励广告换提示、外观付费；严格禁止资源/跳关付费 |
